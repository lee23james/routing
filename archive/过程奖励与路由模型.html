1. 你当时想解决的核心问题

你关注的是：现有 stepwise model routing 大多只看最终结果，或者依赖比较脆弱的启发式信号，但“路由”本质上是一个过程决策问题。

我们当时总结过两类主流方法的局限：

第一类：training-free 路由

比如基于熵、置信度、token 不确定性之类的信号。

问题在于：

  • 往往是一套统一标准套所有场景

  • 阈值敏感，迁移到新任务/新模型时常常要重新调

  • 它们只是在“猜这一步难不难”，但没有直接学习什么叫好的路由轨迹

第二类：RL 训练 router

比如类似 TRIM 那种，让 router 在每一步决定调用大模型还是小模型。

问题在于：

  • 通常主要依赖终局奖励

    • 最终答对了没

    • 总成本高不高

  • 但路由本身是一个序列决策过程

  • 很多错误其实在中间某一步就已经埋下了

  • 只靠终局 reward，信用分配会很难
    也就是：到底是哪一步路由决策出了问题，不够清楚

所以你当时的核心直觉是：

既然 routing 的目标不只是“最后答对”，而是“在合适的地方用合适的模型”，那就应该给路由过程本身打分，而不只是看最后结果。

2. 我们当时提出的核心方向

核心就是这句话：

把过程奖励（process reward）引入 stepwise model routing。

更具体一点：

  • 不只给 router 一个最终 reward

  • 还要对每一步的路由选择是否合理进行评价

  • 让模型学到：

    • 简单步骤用大模型是浪费

    • 困难步骤用小模型容易导致后续推理崩掉

    • 有些步骤虽然暂时没错，但已经在积累风险

    • 有些步骤应该“保守一点”，有些步骤可以“激进一点”

这就是你后来一直在说的那种 rubric 式判断：

  • easy step + choose LRM = waste

  • hard step + choose SRM = risky / wrong

  • 前面某步选错，后面即使形式上还在推，也已经偏了

  • 局部省算力不能以破坏全局正确性为代价

3. 为什么“过程奖励”在 routing 里是合理的

这是我们当时反复强调的一点：
routing 不是普通的答案生成，它更像调度/控制问题。

router 的每一步动作会影响：

  • 当前这一步的生成质量

  • 后续上下文状态 hth_tht​

  • 未来步骤的难度

  • 最终总 FLOPs / latency / accuracy

所以一个路由动作的好坏，不该只由最后答没答对决定，而该由两部分共同决定：

结果层面

  • 最终答案是否正确

  • 总成本是否低

过程层面

  • 这一步是否选对了模型规模

  • 这一步是否造成了可避免的浪费

  • 这一步是否把后续推理带入高风险状态

  • 这一步是否在“该稳的时候稳，该省的时候省”

这就是你当时想把 outcome reward + process reward 结合起来的原因。

4. 当时你最感兴趣的做法：引入 rubric

我们当时其实已经把方向逐渐收敛到：

借鉴 RLCER / Open-Rubric 一类思想，用 rubric 来给 routing 过程打分。

也就是，不是手工只写一两条规则，而是建立一套“评价路由轨迹”的 rubric 体系。

这些 rubric 不是评判“答案写得好不好”，而是评判：

这条 routing trajectory 合不合理。

5. 这些 routing rubric 大概长什么样

我们之前讨论过，rubric 可以分成两层。

A. 局部 step-level rubric

评估单步路由决策是否合理。

例如：

  1. 简单步骤不应调用大模型

    • 如果当前步骤主要是格式整理、显式抄写、简单算术、直接表项抽取

    • 用 LRM 属于资源浪费

  2. 困难步骤不应交给小模型

    • 如果当前步骤涉及复杂组合、隐式约束、多跳表操作、逻辑转折

    • 用 SRM 风险很高

  3. 信息依赖复杂时应更保守

    • 当前步骤如果强依赖前文多个中间变量

    • 或者依赖表格多个字段的联合理解

    • 更适合用 LRM

  4. 错误传播风险高的步骤应优先保证正确性

    • 某一步一旦错，后面都会建立在错的中间状态上

    • 这种步骤应优先选更强模型

  5. 重复/模板化步骤可交给 SRM

    • 如果当前步骤和前面已有 pattern 非常类似

    • 小模型可能足够

B. trajectory-level rubric

评估整条 routing 轨迹是否合理。

例如：

  1. 整体资源分配是否均衡

    • 有没有在大量简单步骤上无意义调用 LRM

  2. 关键转折点是否调用了足够强的模型

    • 比如第一次做复杂聚合、第一次处理歧义、第一次多步推断时

  3. 是否出现“前松后紧”或“前紧后松”的失衡

    • 前面该用大模型没用，后面再补救已经太晚

    • 或者前面全程重型模型，后面其实没有必要

  4. 是否兼顾了准确率与成本

    • 不是一味省，也不是一味稳

  5. 路由策略是否与问题难度匹配

    • 简单题应该更节省

    • 难题应该允许更多 LRM 调用

7. 当时我们还特别讨论过：过程奖励不能只看当前对错

这是个关键点。

你当时已经意识到：

有些 step 当下看起来没错，但其实路由已经选坏了，只是错误还没显现出来。

所以过程奖励不能只问：

  • “这一步生成对了吗？”

还要问：

  • “这一步是不是应该用更强模型？”

  • “这一步虽然暂时生成出来了，但不确定性很高，是否在制造后续风险？”

  • “这一步的选择是否符合全局成本-准确率平衡？”

也就是说，过程奖励评估的是 routing quality，不只是 local answer correctness。

8. 我们当时设想的一个完整 pipeline

这是之前讨论过的比较完整的版本。

Stage 1：收集路由轨迹数据

对每个样本，跑出多种 trajectory，例如：

  • 全 SRM

  • 全 LRM

  • router 当前策略采样出的 mixed trajectory

  • 人工构造一些“明显浪费”或“明显冒险”的轨迹

每条轨迹记录：

  • 每一步上下文

  • 当前步骤内容

  • router 动作

  • 当前调用模型

  • 当前 step 输出

  • 最终正确性

  • 总 FLOPs / latency

Stage 2：构造 rubric

初始 rubric 可以来自：

  • 人工写的高层原则

  • 大模型根据“好路由 / 坏路由”案例归纳

  • 借鉴 Open-Rubric / self-evolving rubric 的方式自动提出

比如初始 rubric 可能是：

  • 简单步骤调用 LRM 扣分

  • 高风险步骤调用 SRM 扣分

  • 关键推理节点调用 LRM 加分

  • 重复模式步骤调用 SRM 加分

  • 总体过度保守或过度激进扣分

Stage 3：训练或调用一个 rubric evaluator

输入：

  • 当前问题

  • 当前 step 上下文

  • 当前动作

  • 当前 step 输出

  • 可选：后续影响摘要

输出：

  • step-level score

  • trajectory-level score

  • 或自然语言 critique

这样就把“路由是否合理”转成可用监督信号了。

Stage 4：把 rubric score 转成过程奖励

可以是：

  • 每一步一个 scalar reward

  • 或轨迹级 reward 再分摊到关键步骤

  • 或 combined reward

例如：

rtproc=w1⋅necessity_score+w2⋅efficiency_score+w3⋅risk_control_scorer_t^{\text{proc}} = w_1 \cdot \text{necessity\_score} + w_2 \cdot \text{efficiency\_score} + w_3 \cdot \text{risk\_control\_score}rtproc​=w1​⋅necessity_score+w2​⋅efficiency_score+w3​⋅risk_control_score

Stage 5：RL 训练 router

可以继续用 policy gradient / PPO / GRPO 一类方法，让 router 学习在每一步做更优决策。

Stage 6：rubric 自演化

这是你最感兴趣的地方之一。

类似 RLCER 的想法：

  • 周期性分析当前 router 常见错误

  • 发现已有 rubric 覆盖不到的新 failure mode

  • 用更强模型总结并补充 rubric

  • 保留高质量 rubric，淘汰低质量 rubric

于是 rubric 不再是静态规则，而会随着训练逐渐变好。

9. 当时我们也分析过 novelty 在哪里

你担心过这个 idea 会不会只是“把 rubric 搬过来”。

我们当时认为，如果做得好，novelty 其实还可以，主要体现在三点：

第一，问题对象不同

RLCER/Open-Rubric 主要是在做 CoT 质量监督。
你这里做的是 routing policy 监督。

这两者不一样：

  • 前者问“这段推理写得好不好”

  • 后者问“这一步该不该调用大模型”

这本身就是不同任务。

第二，奖励对象从结果扩展到资源分配行为

你这里不仅关心正确性，还关心：

  • 资源是否浪费

  • 风险是否被正确控制

  • 决策时机是否合理

这个更接近 cost-aware sequential control。

第三，可以提出“routing-specific rubric taxonomy”

如果你能系统定义一套：

  • waste

  • under-routing

  • delayed escalation

  • over-conservative routing

  • error-propagating routing

这种 routing 特有错误分类，那么贡献会更明确。

10. 当时我们也分析过 feasibility 风险

我们并没有觉得这条路一定稳，因为确实有几个难点。

难点 1：rubric evaluator 的开销

如果每一步都调用一个大模型给 rubric score，成本会很高。

所以后来我们倾向于：

  • 训练时用强 evaluator

  • 推理时只保留 router

  • rubric 模块只用于训练，不参与在线推理

这点很重要，否则系统整体开销会爆炸。

难点 2：过程奖励可能 noisy

rubric 打分本身可能不稳定，尤其是“这一步到底算简单还是困难”常有模糊性。

缓解方式：

  • 先做 coarse-grained rubric，不要一开始太细

  • 用 pairwise preference 替代绝对分数

  • 做 consistency filtering

  • 让 trajectory-level judgment 辅助 step-level judgment

难点 3：怎么定义“正确的局部路由”

因为同一道题可能有多条合理轨迹，并不是唯一最优。

所以我们当时说过：

不要把过程奖励设计成“唯一标准答案”，而要设计成“偏好信号”。

也就是：

  • 这条轨迹比那条更省

  • 这条轨迹比那条更稳

  • 这一步比另一步更合理

用偏好学习往往比硬标签更靠谱。

11. 你当时的一个很重要的判断

你其实已经抓住这个方向最关键的研究价值了：

我们的目的不只是让模型答对，而是想知道“什么样的路由是好的”。

这句话很关键。

因为如果只是答对，那很多 route 都能答对：

  • 全 LRM 也能答对

  • 某些瞎蒙路线也可能答对

但它们不一定是“好路由”。

你想研究的是：

  • good routing policy 的结构

  • 高质量路由的评价准则

  • 如何用这些准则来训练 router

这比单纯追最终准确率更有方法论价值。

12. 当时我们收敛出的推荐方向

如果你真做这个方向，我们之前更倾向于你走这条路线：

一个比较稳的版本

先不要一上来就做 fully self-evolving、超复杂的系统。
先做：

  1. 定义 routing-specific rubric taxonomy

  2. 用强模型给路由轨迹打分

  3. 构造 process reward

  4. 和 final reward 一起训练 router

  5. 验证：

    • accuracy 更高

    • cost 更低

    • credit assignment 更好

    • 策略更可解释

再往后扩展

再加入：

  • rubric self-refinement

  • rubric generation

  • rubric selection / pruning

  • OOD transfer of rubric

这样整条线更自然。

1. 怎么保证 rubric 本身是合理的

核心答案是：不能把 rubric 当成“拍脑袋写的规则”，而要把它做成一个“可验证、可对照、可迭代修正”的对象。

也就是说，rubric 的合理性不是靠“我觉得这个规则不错”来证明，而是靠下面三层来建立。

第一层：把 rubric 限制为“可操作的评价维度”，而不是抽象口号

你不要一上来写这种：

  • 路由要合理

  • 要平衡准确率和效率

  • 要在关键地方更谨慎

这种太空了，judge 很容易漂。

你应该把 trajectory-level rubric 写成少数几个可观察维度。例如：

  1. Efficiency waste
    这条轨迹是否在明显低风险、机械性的阶段频繁调用 LRM，导致不必要的高成本。

  2. Delayed escalation
    在已经出现明显不确定性、分支选择、复杂组合推理信号后，是否仍然迟迟不升级，导致错误积累。

  3. Redundant rethinking
    是否出现多次高成本重算/重想，但这些额外调用没有实质减少错误风险，只是在重复消耗资源。

  4. Error attribution quality
    若最终失败，失败更像是：

    • 早期路由失误导致的级联错误

    • 中途升级太晚

    • 还是即使使用 LRM 也无法解决的能力边界

  5. Global allocation quality
    整体上是否把昂贵计算预算集中在真正关键的转折点，而不是平均撒胡椒面。

这种 rubric 就比较接近 OpenRS 里那种“显式、可检查的原则”，而不是黑盒单分数。OpenRS 的一个重要出发点就是，单一 opaque scalar reward 容易脆弱和被 hacking，应该把原则显式化，并把一些部分做成 verifiable guardrails。arXiv+1

第二层：rubric 不是直接“发明”，而是从对比案例里归纳出来

最稳的做法不是先写规则，再相信它；而是先构造成对轨迹，再从中抽象 rubric。

例如同一道题，你收集几条轨迹：

  • A：早期就用 LRM，成本高但很稳

  • B：前面一直 SRM，关键点升级晚了，最后错

  • C：答对了，但中间三次重复 rethinking，成本很高

  • D：答对了，且只在两个关键节点升级

然后问强 judge 或专家：

  • A 和 D 哪条更好，为什么

  • C 和 D 哪条更好，为什么

  • B 的失败更像哪种原因

从这些 pairwise preference 的“为什么”里抽取共性，再形成 rubric。
这比先拍脑袋写 rubric 可靠得多。

本质上，你是在做：

rubric ← 对大量轨迹偏好解释的压缩总结

而不是：

偏好判断 ← 生硬套已有 rubric

前者稳很多。

第三层：要求 rubric 有“外部效度”

rubric 合不合理，最后不是看它写得漂亮，而是看它是否能解释和预测真正重要的现象。

一个合理 rubric 至少要满足三件事：

1) 区分度

它要能把“看起来都答对”的轨迹分开。
例如同样答对，能区分：

  • 高效且关键点升级恰当

  • 全程滥用 LRM

  • 靠反复 rethinking 硬堆出来

如果分不开，那 rubric 没价值。

2) 因果相关性

高 rubric 分的轨迹，应该更接近你真正想要的行为：

  • 更低成本下保持正确

  • 更少无效升级

  • 更少无意义重复思考

  • 失败时更可归因

3) 泛化性

它不能只适配某个数据集的一种表面模式。
如果换一类题、换模型对、换 budget 设定，rubric 完全失灵，那说明规则学的是表面相关，不是本质。

RLCER 一类工作会强调 rubrics 要随着训练演化，否则会和模型行为分布脱节；这恰好说明“rubric 合理性”本身就不是一次性静态给定的，而需要持续校正。arXiv+1

2. 怎么保证“这条轨迹符合 rubric 的判断”是正确的

这个问题其实比前一个更难。
因为就算 rubric 本身写得不错，judge 在具体判某条轨迹时仍然可能误判。

这里不能追求“绝对正确”，你要追求的是：高一致性、高校准、低系统性偏差。

最稳的方法是把“轨迹判定”拆成三类信号，而不是只靠一个 LLM judge 直接打总分。

第一类：可验证信号

能程序化验证的部分，尽量不要让 judge 主观判断。

例如：

  • 总 LRM 调用次数

  • 总 FLOPs / token cost

  • rethinking 次数

  • 某一步后答案状态是否已经偏离 gold rationale / verifiable intermediate state

  • 升级发生在第几个 step

  • 升级后错误是否被纠正

这些都可以直接算。

所以像“资源浪费”“重复 rethinking”这种，最好做成半可验证项，不要完全交给 judge 主观打分。
这也和 OpenRS 里 pointwise verifiable rubrics 的思路一致：能验证的部分尽量程序化，用作 guardrail。arXiv+1

第二类：归因信号

最难的是失败归因：
到底是“路由错了”，还是“模型能力不够”。

这个不能只看最终错没错，而要做counterfactual 检查。

也就是对失败轨迹，在关键节点上做最小反事实实验：

  • 把第 ttt 步的 SRM 改成 LRM，后面保持一致或局部重滚

  • 看错误是否被消除

  • 再测第 t+1t+1t+1 步、第 t+2t+2t+2 步

如果某个早期替换能显著修复后续，那失败更可能归因于早期 under-routing。
如果把关键嫌疑步都升级为 LRM 仍然失败，那更可能是task/model capability limit，不是路由器的问题。

这一步很重要，因为它能把你的第二个 concern 从“judge 猜原因”变成“用反事实证据支持归因”。

你甚至可以把 failure attribution rubric 明确写成：

  • Routing-induced failure：存在早期关键 step 的升级反事实能修复轨迹

  • Late-escalation failure：升级能修复，但必须在更早阶段介入

  • Capability-bound failure：即使关键可疑步全部升级，仍失败

  • Wasteful-success：成功，但对比更省轨迹存在明显冗余

这样 judge 就不是空口判断，而是在看证据。

第三类：偏好判断信号

在仍然无法完全程序验证的地方，不做 pointwise absolute score，而做 pairwise preference。

也就是不问：

  • “这条轨迹打几分？”

而问：

  • “在同一道题的这两条轨迹里，哪条更好，为什么？”

因为 pairwise 一般比绝对打分稳定，OpenRS 也强调 pairwise adaptive rubric 比单点加权标量更有辨别力。arXiv

对你这里很合适，因为很多时候“单条轨迹是否好”不好说，但“两条轨迹谁更优”相对容易：

  • 都答对时，比效率和是否有冗余

  • 都答错时，比谁的失败更可避免

  • 一对一错时，比成功是否建立在明显浪费之上，以及失败是否其实不可归咎于路由

3. 所以一个更稳的设计是：不要直接学“打分”，先学“比较”和“归因”

这是我觉得你现在最应该收紧的地方。

你不要先说：

我们设计一个 trajectory rubric，然后 judge 给每条轨迹一个总分。

这太脆了。

你应该说：

我们先把 trajectory-level supervision 拆成三类更可靠的信号：

  1. verifiable cost/waste signals

  2. counterfactual failure attribution

  3. pairwise preference under explicit rubric
    最后再把它们聚合为 trajectory reward。

这样你那两个担心会大幅减轻。

一、不要让模型“凭空写 rubric”，而要让它“从轨迹对比中归纳 rubric”

这是最核心的一点。

你的任务不是一般的 response evaluation，而是 routing trajectory evaluation。
所以 rubric 的输入不应该只是一个问题，而应该是：

  • 题目 xxx

  • 一组候选路由轨迹 τ1,τ2,...\tau_1, \tau_2, ...τ1​,τ2​,...

  • 每条轨迹的结果信息：

    • 最终是否答对

    • 总成本 / LRM 调用次数 / token 开销

    • rethinking 次数

    • 关键步骤位置

    • 可选的反事实修复结果

然后让模型做的不是：

“请直接生成 rubric”

而是先做：

“比较这些轨迹，解释为什么 A 比 B 更好，或为什么 B 的失败更可避免。”

再从大量这种对比解释里抽象出 rubric。

这一步非常重要，因为它把 rubric 的来源变成了经验归纳，而不是主观脑补。
这和 OpenRubrics 的 contrastive rubric generation 本质上是一致的，只不过你这里的对比对象从“两个答案”换成了“两个 routing trajectories”。arXiv+1

二、我建议你生成两层 rubric，而不是一层

第一层：Meta-rubric

这是“宪法”，是比较稳定的上层原则，不直接针对某个具体样本。

你这里的 meta-rubric 可以非常少，只保留 4–6 条，比如：

  1. Efficiency over unnecessary strength
    不鼓励在低风险、机械步骤上滥用 LRM。

  2. Timely escalation at critical uncertainty
    在关键分支、复杂组合、错误传播风险高的阶段，应及时升级。

  3. Avoid redundant rethinking
    不鼓励高成本但无实质增益的重复思考或重复重算。

  4. Prefer unavoidable-failure awareness
    若失败不可由更优路由修复，则不应把责任全部归于 router。

  5. Global allocation matters more than isolated thrift
    路由优劣看整体资源分配质量，而不是某一步是否省了一次调用。

这套东西要尽量稳定，不要频繁变。

这就对应 OpenRS 里讲的 explicit meta-rubric：先有上层规范，再在具体样本上实例化，而不是每次都自由发挥。arXiv+1

第二层：Instance-adaptive trajectory rubric

针对某道题、某组轨迹，模型基于 meta-rubric 动态生成更具体的评价维度。

比如在某个样本上，生成出来的是：

  • 该题前两步主要是表项定位，频繁调用 LRM 属于 waste

  • 第三步首次出现多条件聚合，是关键升级点

  • 第四、五步的 repeated LRM calls 没有显著修正前面状态，属于 redundant rethinking

  • 该失败若在第三步升级可被修复，因此更接近 delayed escalation than capability limit

这类 rubric 不需要长期保存为全局规则，但非常适合做当前样本或当前 batch 的 judge 依据。

这本质上就是把 OpenRS 里的 pairwise adaptive rubric 迁移到 routing 场景。arXiv+1

三、生成 rubric 时，输入里一定要包含“正反对比轨迹”

这是我最强烈建议你加的。

如果你只给模型看一条轨迹，让它自己想 rubric，它大概率会产出泛泛之谈。
但如果你给它看同一道题的几条不同风格轨迹，比如：

  • Efficient success

  • Wasteful success

  • Routing-induced failure

  • Capability-bound failure

然后让它回答：

  1. 哪条更优，为什么

  2. 哪些维度真正区分了它们

  3. 哪些判断可以泛化到别的题

模型生成的 rubric 会扎实很多。

我甚至建议你专门构造一批“教学型对比集”来生 rubric，例如：

  • 同样答对，但一个明显更浪费

  • 同样答错，但一个是早期 under-routing，一个是 capability limit

  • 都高成本，但一个升级点合理，一个是反复 rethinking

这样模型更容易学出你真正想要的 trajectory taxonomy。

四、rubric 的结构最好写成“硬规则 + 原则”两部分

这个可以直接借 OpenRubrics。它把 rubric 分成 hard rules 和 principles，这个分法对你特别合适。arXiv+1

1. Hard rules

这些是尽量可验证、可落地的。

例如：

  • 若两条轨迹最终都正确，则更少 LRM 调用、且无明显重复 rethinking 的轨迹优先

  • 若轨迹失败且某关键早期 step 升级为 LRM 后可修复，则失败应归因为 routing-induced failure

  • 若将关键可疑 step 升级为 LRM 仍无法修复，则不应把失败主要归咎于 router

  • 对于同一题，若一条轨迹的额外 LRM 调用未显著提升中间状态质量或最终结果，则判为 wasteful

2. Principles

这些是更软的偏好导向。

例如：

  • 预算应集中在错误传播风险最高的节点

  • 不应以表面节省换取高概率的后续崩塌

  • 不应以反复重想来掩盖早期错误决策

  • 成功的轨迹也要考虑资源配置质量，而不是只看 final correctness

这种二分法的好处是：
hard rules 提供稳定锚点，principles 提供泛化空间。

五、不要直接接受模型第一次生成的 rubric，要做两轮筛选

第一轮：一致性筛选

这个几乎可以照搬 OpenRubrics 的思路。
让模型生成 rubric 后，立刻用这套 rubric 去判它生成时参考的轨迹对，检查它是否能推出与你已有偏好标签一致的结论。不能一致的，直接丢弃。OpenRubrics 就是用这种 preference-label consistency 加 rejection sampling 来降低噪声。arXiv+1

你这里可以变成：

  • 给出轨迹对 (τa,τb)(\tau_a, \tau_b)(τa​,τb​)

  • 已知偏好或弱标签，比如 τa≻τb\tau_a \succ \tau_bτa​≻τb​

  • 让 rubric-judge 用新生成 rubric 重判

  • 若判不出或判反了，则此 rubric 视为低质量

这能过滤掉大量“写得好看但不判事”的 rubric。

第二轮：外部效度筛选

这一步是防止 rubric 学歪。

你保留的 rubric，应该在 held-out 轨迹集上表现出以下性质：

  • 能把 efficient success 和 wasteful success 分开

  • 能把 routing-induced failure 和 capability-bound failure 分开

  • 用 rubric 得到的轨迹偏好，与真实反事实修复结果有较高相关性

  • 高 rubric 分的轨迹，平均 accuracy-cost tradeoff 更好

如果做不到，说明这套 rubric 虽然内部一致，但没有真正抓住你想要的 routing quality。

六、我建议你把“rubric 生成”和“rubric 演化”分开

阶段 A：离线生成初始 rubric 库

先用强模型离线生成一批 trajectory-level rubrics。
来源是高质量对比轨迹 + 轨迹偏好解释。

这一步先追求稳，不追求自动化极致。

阶段 B：训练中做轻量演化

再借鉴 RLCER 的 self-evolving 思路，对 rubric 库做周期更新。RLCER 的一个核心观点就是：静态 rubric 会和不断变化的策略分布脱节，所以 rubric 需要自我演化。arXiv+1

但你这里不要让演化太自由。
建议只在以下场景触发：

  • 发现大量 judge 不确定样本

  • 出现新的失败模式，现有 rubric 无法解释

  • 某些 rubric 在 held-out 上区分度明显下降

然后再让强模型：

  1. 总结现有 rubric 的覆盖盲区

  2. 提出候选新 rubric

  3. 经过一致性 + 外部效度筛选后加入库中

  4. 淘汰低使用率或低稳定性的 rubric

这会比“每轮都自由生成新 rubric”安全很多。

七、具体到你的 routing 任务，我最推荐的 rubric 获取流程

我给你一个最像方法部分的版本。

Step 1：构造轨迹池

对每个问题生成多条轨迹：

  • 全 SRM

  • 全 LRM

  • 当前 router 采样轨迹

  • 随机温度扰动轨迹

  • 针对关键节点做局部升级/降级的反事实轨迹

Step 2：给轨迹打弱标签

不是人工打分，而是自动得到一些 weak signals：

  • final correctness

  • cost / LRM count

  • rethinking count

  • counterfactual repairability

  • 关键节点是否存在 delayed escalation

  • 是否属于 avoidable failure

Step 3：构造对比样本

把同题的轨迹组成 pair 或 small set，例如：

  • efficient success vs wasteful success

  • routing-induced failure vs capability-bound failure

  • delayed escalation failure vs timely escalation success

Step 4：让强模型生成“轨迹偏好解释”

不是先生成 rubric，而是先解释：

  • 哪条更好

  • 为什么

  • 关键区分点是什么

  • 哪些是可以泛化的原则

Step 5：从解释归纳 rubric

再让模型把这些解释压缩成：

  • meta-rubric candidates

  • instance-adaptive rubric templates

  • hard rules / principles

Step 6：做两阶段筛选

  • consistency filtering

  • held-out external validation

Step 7：形成 rubric bank

保留少量高频、高稳定性、高判别力的 rubric，作为 judge 的基础规范

Step 8：训练中周期演化

定期针对新 failure modes 增量更新 rubric bank

八、如果你只想先做一个最小可行版本

那我建议你先不要做 full self-evolving。
先做这个简化版：

  1. 用强模型比较同题轨迹对，生成 preference explanations

  2. 从解释里总结出一小组 trajectory-level meta-rubrics

  3. 把 rubric 分成 hard rules 和 principles

  4. 用 consistency filtering 保留可靠 rubric

  5. judge 时优先做 pairwise comparison，不做脆弱的单轨迹绝对打分

这个版本已经足够有内容了，而且比“直接让模型写 rubric”靠谱很多。

6. 那高质量 rubric 的标准到底是什么

我建议你明确提出三条标准，作为 rubric 质量定义。

1. Fidelity

它是否真的反映 routing quality，而不是 response style 或表面结果。

例如：

  • 答对但极度浪费，应被判得没那么高

  • 答错但属能力边界，不应被过度惩罚

这就是 fidelity。

2. Coverage

它是否覆盖主要成功/失败模式，而不是只对某一类样本有效。

例如：

  • wasteful success

  • delayed escalation

  • avoidable failure

  • unavoidable failure

  • redundant rethinking

至少要覆盖这些。

3. Reliability

它是否在不同 judge、不同样本、不同轨迹对比中保持稳定。

例如：

  • 同一条轨迹重复评估结论一致

  • 不同 phrasing 下判断差别不大

  • 在 held-out 样本上还能区分优劣

这就是 reliability。

让 rubric judge 受证据约束，而不是自由发挥。

  • 局部反事实轨迹：把某些 accept 改成 regenerate，或把某些 regenerate 改成 accept

A3. 给每条轨迹提取结构化证据

这一步必须程序化，不能全靠 judge 主观判断。
为每条轨迹记录：

  • final correctness

  • total LRM tokens / estimated cost

  • number of regenerate actions

  • regeneration positions

  • repeated regeneration / rethinking count

  • early-error repairability

  • key-step counterfactual repairability

其中最后两项最关键。
做法是：对失败轨迹，找到若干“嫌疑关键步”，把这些步从 accept 改成 regenerate，重新 roll out 后续，观察最终是否被修复。
如果某个早期改动能显著修复失败，就说明这是 routing-induced failure；如果关键嫌疑步都升级了仍失败，更接近 capability-bound failure。这和 TRIM 关注 cascading failures 是一致的，只不过你把“是否可修复”显式化成归因证据。arXiv+1

A4. 动态生成 candidate rubrics

这里借 RLCER 的“self-proposed rubric”思想，但只做成辅助模块，不把它当主贡献。RLCER 的核心是：rubrics 可以由模型提出，并作为 outcome reward 之外的额外监督。arXiv+1

具体输入不是单个问题，而是：

  • 问题 xxx

  • 同题的 3–5 条代表性轨迹

  • 每条轨迹的 correctness / cost / repairability 证据

  • 固定 meta-rubric

让强模型输出两类内容：

  • Hard rules

  • Principles

这点直接借 OpenRubrics。OpenRubrics 明确把 rubric 拆成 hard rules 和 principles，并用 preference-label consistency 过滤噪声。arXiv+1

你这里的 candidate rubric 示例：

Hard rules

  • 若两条轨迹都正确，优先成本更低且无显著冗余 regenerate 的轨迹

  • 若失败轨迹在早期关键步升级后可修复，则判为 routing-induced failure

  • 若关键嫌疑步升级后仍失败，则不应主要惩罚 router

Principles

  • 预算应集中在高风险转折点

  • 不应通过多次高成本重想弥补早期错误

  • 成功轨迹也应根据资源分配质量区分高下

A5. 筛选 rubric

这一步决定你方法靠不靠谱。

第一层：consistency filtering

直接借 OpenRubrics：
用生成出的 rubric 去重判构造它时参考的轨迹对，看是否能复现已知弱偏好；不一致就丢弃。arXiv+1

第二层：evidence compatibility filtering

直接借 OpenRS 的思路：
让 rubric judgment 必须和程序证据相容。OpenRS 强调 pointwise verifiable rubrics 作为可验证 guardrails。arXiv+1

比如：

  • 明显更贵、且无准确率收益的轨迹不能被评成更优

  • 可被早期升级修复的失败不能被评成 capability-bound

  • 反复 regenerate 但无收益的轨迹不能被评成“高质量成功”

经过这两层后，得到一个小型 rubric bank。

五、训练算法怎么选

第一版直接用 PPO，不要改。

  1. 用当前 router rollout 一批 trajectories

  3. 调用 rubric judge / programmatic checks 得到 process reward

  4. 合成 total reward

  5. 用 PPO 更新 router

七、落地实现时最容易踩坑的地方

1. 不要让 rubric 直接决定每一步动作

  • 每一步都调用 rubric judge

  • 再反过来指导 accept/regenerate

这会非常慢，也非常 noisy。
最稳的是 episode-level reward shaping。

2. 不要完全依赖 LLM judge

  • token cost

  • regenerate count

  • repeated rethinking

  • counterfactual repairability

LLM judge 只负责：

  • 全局 allocation quality

  • delayed escalation 的解释性判断

  • success/failure 类型归纳

先做：

  • 固定 meta-rubric

  • 小型 rubric bank

  • 周期更新

这已经足够了。

4. 不要用绝对分，优先用 pairwise preference

OpenRS 的一个核心点就是 criterion-wise pairwise judging 更稳。你这里也一样。arXiv+1

十、最关键的实验怎么做

你要证明的不是“rubric 很聪明”，而是“process reward 让 routing 学得更好”。

所以核心实验应该是：

1. 主结果

和 TRIM-RL 比：

  • same cost 下更高 accuracy

  • same accuracy 下更低 cost

2. 行为分析

证明你真的学会了更好的 routing：

  • fewer redundant regenerations

  • fewer delayed escalations

  • fewer avoidable failures

  • better allocation on critical steps

3. 归因分析

把失败分成：

  • routing-induced

  • capability-bound
    看你的方法是否主要减少前者，而不会无脑惩罚后者

4. Ablation

去掉：

  • waste penalty

  • avoidable-failure penalty

  • rethinking penalty

  • pairwise rubric judge
    分别看性能变化

一句话总结你的落地方案：

保留 TRIM 的 stepwise PPO 框架不变，用“程序证据 + rubric judge”在 episode 结束后给 trajectory 追加一个过程奖励，显式奖励高效成功、惩罚浪费成功与可避免失败、弱化对能力边界失败的惩罚，从而把 outcome-only routing RL 升级为 trajectory-aware routing RL。 arXiv+3arXiv+3arXiv+3
